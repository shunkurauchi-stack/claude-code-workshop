STEP 1

まず CLAUDE.md を書く

CLAUDE.md はプロジェクトのルートに置くマークダウンファイル。
Claude Code を起動するたびに自動で読み込まれ、
毎回説明しなくてもAIが文脈を理解した状態になる。

🗂️

何を書くか

プロジェクト概要・技術スタック・コード規約・ディレクトリ構成・よく使うコマンド・やってはいけないこと

👥

チームで共有する

Git にコミットしてチーム全員で使う。全員のClaude Codeが同じ前提で動くようになる

CLAUDE.md の例
CLAUDE.md # My App — Claude Code 設定

## プロジェクト概要
FastAPI + React で作る在庫管理システム

## 技術スタック
- Backend: Python 3.12 / FastAPI / SQLAlchemy
- Frontend: React 18 / TypeScript / Tailwind CSS
- DB: PostgreSQL 16

## コード規約
- Python: 型ヒント必須、docstring は Google スタイル
- API エラーは AppException を継承したクラスで raise
- テストは pytest。fixture は conftest.py に集約

## よく使うコマンド
```bash
make dev # 開発サーバー起動
make test # テスト実行
```

## やってはいけないこと
- 生 SQL は原則禁止（ORM を使う）
- .env ファイルを Git にコミットしない

STEP 1

CLAUDE.md を育てるコツ

✅

書くべき内容

プロジェクトの目的・背景
使っているライブラリ・バージョン
命名規則・フォルダ構成のルール
AIに「知ってほしい罠」（既知のバグ・回避策）
よく使う CLI コマンド

🚫

書かなくていい内容

コードやコメントで読めばわかること
パスワード・APIキーなどの機密情報
長すぎる説明（簡潔さが重要）
Git 履歴で追えること
一時的なメモ（作業ログなど）

✨
/init コマンドで自動生成させよう： コードが存在する状態で /init と打つとコードベースを読み込み、CLAUDE.md の雛形を自動生成してくれる。
完全に新規の場合は npm create / django-admin startproject などでスキャフォールドしてから実行する。そこから手直しするのが一番早い
💡
Claude Code はセッションをまたいで記憶がリセットされる。「この説明また必要だな」と感じたことは CLAUDE.md に書いておくと次回から不要になる。書けば書くほど的外れな提案が減っていく

STEP 2

設計・要件整理をAIと一緒にやる

実装を始める前に、Claude Code と
「何を作るか」を言語化するのが重要。
この段階で曖昧さをなくすほど実装がスムーズになる。

設計フェーズの指示例
ユーザーが商品を検索してカートに追加できる機能を実装したい。
まず実装方針・必要なAPIエンドポイント・DBスキーマの変更点を整理して提案してほしい。コードはまだ書かなくていい。
Claude Code の回答（例）
提案する実装方針：
① GET /api/products?q= で検索API
② POST /api/cart/items でカート追加
③ cart_items テーブルを新設（user_id, product_id, quantity）
④ 既存の Product モデルに stock_count を追加…
設計フェーズでやること
🗺️

実装方針のレビュー

「こう実装しようと思うが問題ないか確認して」と相談。見落としを早期発見

🔍

既存コードへの影響調査

「この変更が影響するファイルをすべて洗い出して」で影響範囲を把握

📊

DBスキーマ設計のレビュー

ER図やスキーマ案を提示して「正規化の問題・インデックスの漏れ」を確認

⚠️

リスクの洗い出し

「この実装のリスクや落とし穴を教えて」と聞いておくと後で後悔しない

STEP 3

実装フェーズ ─ 効果的な指示の出し方

悪い指示 vs 良い指示
❌ 悪い指示
カート機能を実装して
✅ 良い指示
backend/routers/cart.py に以下のエンドポイントを追加して：
• POST /cart/items — リクエスト: {product_id, quantity}
• 在庫が不足の場合は 400 エラーを返す
• 既存の ProductService のスタイルに合わせて実装
• 実装後にユニットテストも追加して
💡
指示に含めると精度が上がる3点：
① どのファイルに　② 何を　③ どんな制約で
実装を小さく分割する
1

1機能 = 1会話

「カート追加」「在庫チェック」「決済連携」をまとめて頼まない。一つずつ動作確認してから次へ

2

既存スタイルを参照させる

「src/services/user.py と同じスタイルで」と指定するだけで命名・構造が揃う

3

途中で確認する

「まずファイル構成の案だけ提示して、OKなら実装して」と2段階にすると安全

4

コンテキスト管理

長くなったら /compact で圧縮して作業継続。それでも混乱するなら /clear で全リセット

STEP 4

テスト・デバッグを効率化する

🧪

テスト自動生成

実装した関数を選択して
「この関数のユニットテストを書いて。エッジケースも含めて」
と指示するだけで pytest / Jest のテストが出てくる

🐛

エラー解析

スタックトレースをそのまま貼り付けて
「このエラーの原因と修正方法を教えて」
コードベース全体を検索して原因を特定してくれる

テスト生成の指示例
backend/services/cart.py の add_item() のテストを書いて。
以下のケースを必ずカバーして：
• 正常系（在庫あり）
• 在庫不足で例外が発生するケース
• product_id が存在しないケース
既存の tests/services/test_product.py と同じ形式で
⚡

テスト実行も任せる

「テストを実行して失敗したら直して」と言えば、実行→失敗確認→修正まで自律的にこなす

🔬

コードレビューを依頼する

「このPRのコードで問題になりそうな点を教えて」でセルフレビューの精度が上がる

📄

ドキュメント生成

「このモジュールのAPIドキュメントをMarkdownで書いて」で仕様書を自動生成

Tips

よくある失敗パターンと避け方

❌

「一発で全部やって」

大きすぎる指示は品質が落ちる。
→ 1機能ずつ、確認しながら進める

特にファイルをまたぐ変更ほど分割が重要

❌

提案をそのままAccept

AIの変更は必ずレビューする。セキュリティ脆弱性・古いAPIの使用・テスト漏れが混入することがある。
→ diff を必ず確認する

❌

文脈なしで質問する

「バグを直して」だけでは的外れな回答になる。
→ ファイル名・行番号・エラー内容を添える

CLAUDE.md があれば省略できる部分も増える

❌

機密情報をプロンプトに貼る

APIキー・パスワード・個人情報は絶対に貼らない。
→ ダミー値や変数名で代替する

.gitignore の確認も事前に忘れずに

💡
黄金ルール：Claude Code は超優秀なジュニアエンジニア。丁寧に文脈を伝えるほど良い仕事をするが、最終責任は人間が持つ
